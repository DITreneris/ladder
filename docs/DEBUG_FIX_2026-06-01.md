### C-03 Wave 1 mistake (do not repeat)

CHANGELOG 1.8.4 had moved `onCoffee` after render so the badge cleared cleanly. Wave 1 (`f83ff9d`) restored that order for animation ‚Äî but **`app.ts` looks up the badge on `.next-rung` before render**, so post-shift render leaves nothing to animate. Correct order for current DOM code: **`onCoffee` ‚Üí `renderRungs()`** with `fillSlot` skip during pickup animation.

## Root cause

### What we measured

Playwright on 390√ó844 viewport, production preview:

| Moment | `#gamePlayArea` width | `#gameContentColumn` width | `#app` width |
|--------|----------------------|------------------------------|--------------|
| Tap 0 (game start) | **283px** | **315px** | **341px** |
| Tap 1 (after first climb) | **193px** | **225px** | **251px** |

After fix (`grid` + stable `#app`):

| Taps 0‚Äì8 | `#gamePlayArea` width |
|----------|------------------------|
| All | **316px** (stable until game-over screen hides play area) |

### Mechanism

1. `index.html` used `body { flex; items-center; justify-center }` with an empty `#app` wrapper.
2. `.cl-phone-shell` inside `#app` is `w-full` ‚Äî circular dependency with parent sizing.
3. On first tap, content reflows (HR memo queue, `#imminentHint`, `renderRungs`, climb animation).
4. Flex layout recomputed `#app` **min-content** width downward (~90px).
5. Everything inside ‚Äî HUD, play area, ladder rails, tap deck ‚Äî scaled down together.

Showing HR memo **alone** (toggle visibility without a tap) did **not** change width. The shrink required the full tap + reflow cycle inside the unstable flex root.

### Fix (files)

- `apps/mini-app/index.html` ‚Äî `grid place-items-center`; `#app` class `w-full max-w-md min-w-0`
- `apps/mini-app/src/style.css` ‚Äî `#app { width: 100%; max-width: 28rem; min-width: 0 }` and `html.cl-in-telegram #app { max-width: none; height: 100% }`

## Ruled out

| Hypothesis | Why rejected |
|------------|--------------|
| HR memo rail steals horizontal space | Memo show/hide alone keeps play width at 283px |
| Old `w-48` (192px) ladder | Removed in v1.8.2; no `w-48` in current source |
| `max-w-[5.5rem]` rung slots | Caps badge columns, not shell width; connectors still span play area |
| Reorg slide / rung advance animation | Vertical transforms only; not the ~90px shell shrink |
| Vertical rung compression | HR memo reduces play **height** and rung row height (40‚Äì52px) ‚Äî real but separate from horizontal bug |
| Game logic / obstacle RNG | Width changed before promotion or rank change |

## Lessons learned ‚Äî save time next time

### 15-minute triage order

1. **Bundle hash** ‚Äî prod HTML vs local build ([DEBUG_ENV_TRIAGE.md](DEBUG_ENV_TRIAGE.md)). Stale prod wastes hours.
2. **Measure width at tap 0 vs tap 1** ‚Äî `#gamePlayArea.clientWidth` and `#gameContentColumn.clientWidth` (use `clientWidth`, wait ‚â•500ms after tap).
3. **`npm run qa:layout`** ‚Äî HUD / play / tap column alignment.
4. **`npm run qa:viewport`** ‚Äî overflow and rung fit.
5. **Separate buckets** ‚Äî layout width vs reaction/emoji vs network/API trust. Do not debug all three as one mystery.
6. **Game logic last** ‚Äî only after layout metrics are stable.

### Measurement pitfalls (this session)

| Pitfall | Effect |
|---------|--------|
| `window.engine` not exposed | Programmatic `handleTap` silently no-op‚Äôd; looked like stable width |
| `getBoundingClientRect` during animations | Noisy widths (283 ‚Üí 192 ‚Üí 322 oscillation) |
| Clicks during death shake | Playwright ‚Äúelement not stable‚Äù false failures |
| Confusing height shrink with width shrink | HR memo shortens rungs; user said ‚Äúsusiaureja‚Äù (narrow) ‚Äî check both axes |

### Recommended guardrails (implemented)

| Guardrail | Script | CI |
|-----------|--------|-----|
| Post-tap width stable | `npm run qa:layout` (`layout-audit.mjs`) | yes |
| Tutorial coffee + meeting death | `npm run qa:coffee` (`coffee-pickup-qa.mjs`, `?qa=1`) | yes |
| Viewport / rung fit | `npm run qa:viewport` | yes |
